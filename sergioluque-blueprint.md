## 1. Project Overview

### Vision

Rediseño completo del sitio web del compositor e investigador Sergio Luque. El nuevo sitio reemplaza el WordPress actual con una plataforma moderna, rápida y bilingüe (ES/EN) que funciona como archivo vivo de su obra: catálogo de composiciones con reproductor de audio propio, proyectos, blog académico y presencia institucional. Todo el contenido es gestionable por el propio Sergio desde Sanity Studio sin tocar código.

El estilo visual está inspirado en la estética de sellos como Karlrecords, Editions Mego y los diseños de álbum de Florian Hecker y Marcus Schmickler: negro puro, tipografía monoespaciada stark, sin colores decorativos, alta densidad de información con mucho espacio en blanco.

### Goals
- Presentar el catálogo completo de obras con reproductor de audio self-hosted
- Dar a Sergio un CMS visual y sencillo para gestionar todas las secciones
- Sitio bilingüe español/inglés con SEO optimizado en ambos idiomas
- Cero dependencias de plataformas externas para el audio (fuera de Cloudflare R2)

### Success Metrics
- Sanity Studio: Sergio puede crear/editar una obra, post o proyecto en menos de 5 minutos
- Lighthouse score ≥ 90 en Performance, Accessibility, SEO
- Todas las páginas sirven en < 1.5s TTFB desde Europa

## 5. API Design

El sitio no tiene una API propia compleja. Los datos vienen de Sanity vía GROQ en Server Components con ISR. El único endpoint custom es el formulario de contacto.

### Routes
| Method | Path | Description | Auth |
|--------|------|-------------|------|
| POST | `/api/contact` | Envía email de contacto via Resend | No |

### Endpoint: POST /api/contact
**Request body:**
```json
{
  "name": "string (required)",
  "email": "string (required, valid email)",
  "subject": "string (required)",
  "message": "string (required, max 2000 chars)"
}
```
**Response:**
```json
{ "success": true }
// o
{ "success": false, "error": "mensaje de error" }
```
**Validación:** Zod schema. Rate limiting básico con headers.

### GROQ Queries principales (en `lib/sanity/queries.ts`)
- `getObras(locale)` — todas las obras ordenadas
- `getObraBySlug(slug, locale)` — obra individual con todos los campos
- `getObrasDestacadas(locale)` — obras con `isFeatured: true`
- `getPosts(locale)` — posts publicados, desc por fecha
- `getPostBySlug(slug, locale)` — post individual
- `getProyectos(locale)` — todos los proyectos
- `getEventos(locale)` — eventos ordenados por fecha
- `getPublicaciones()` — publicaciones académicas
- `getSettings()` — singleton de configuración global

## 6. Frontend Architecture

### Pages / Routes
| Ruta | Página | Descripción |
|------|--------|-------------|
| `/[locale]` | HomePage | Hero (nombre + rol) + obras destacadas + próximos conciertos |
| `/[locale]/obras` | WorksPage | Grid/lista de todo el catálogo con AudioPlayerMini por obra |
| `/[locale]/obras/[slug]` | WorkDetailPage | Obra individual: reproductor full, descripción, info técnica |
| `/[locale]/proyectos` | ProjectsPage | Grid de proyectos |
| `/[locale]/proyectos/[slug]` | ProjectDetailPage | Proyecto individual |
| `/[locale]/blog` | BlogPage | Listado de posts con paginación |
| `/[locale]/blog/[slug]` | PostPage | Post individual con Portable Text renderizado |
| `/[locale]/bio` | BioPage | Biografía larga + foto |
| `/[locale]/publicaciones` | PublicationsPage | Lista de artículos académicos |
| `/[locale]/conciertos` | ConcertsPage | Próximos y pasados |
| `/[locale]/contacto` | ContactPage | Formulario |

### Component Hierarchy — WorksPage
```
WorksPage (Server Component)
  ├── Header (Server)
  │   └── LanguageSwitcher (Client)
  ├── WorkGrid (Server)
  │   └── WorkCard[] (Server)
  │       └── AudioPlayerMini (Client) ← "use client" aquí
  └── Footer (Server)
```

### Component Hierarchy — WorkDetailPage
```
WorkDetailPage (Server Component)
  ├── Header
  ├── WorkDetail (Server)
  │   ├── AudioPlayer (Client) ← reproductor principal
  │   │   ├── WaveformBars (Client)
  │   │   └── PlaybackControls (Client)
  │   ├── WorkMetadata (Server) — año, formación, premiere, encargos
  │   └── WorkDescription (Server) — Portable Text renderer
  └── Footer
```

### State Management
- **Server Components por defecto** para todo lo que es solo visualización de datos.
- **Client Components** únicamente para: AudioPlayer, LanguageSwitcher, ContactForm.
- **ISR** con `revalidate: 3600` en todas las páginas de contenido. Cuando Sergio publica algo en Sanity, un webhook llama `revalidatePath` vía Sanity webhook → Next.js API route.
- No hay estado global (no Zustand, no Context). El reproductor de audio es local al componente.

## 7. Design System

### Colors
| Token | Hex | Uso |
|-------|-----|-----|
| `--bg` | `#0A0A0A` | Fondo de página (OLED black) |
| `--surface` | `#141414` | Cards, paneles, player |
| `--surface-hover` | `#1E1E1E` | Hover en cards |
| `--border` | `#2A2A2A` | Bordes, separadores |
| `--text-primary` | `#F0F0F0` | Texto principal |
| `--text-secondary` | `#888888` | Metadatos, fechas, labels |
| `--text-muted` | `#555555` | Texto de relleno, disabled |
| `--accent` | `#D4D4D4` | Links hover, accents mínimos |
| `--error` | `#EF4444` | Errores de formulario |
| `--success` | `#22C55E` | Confirmaciones |

**Regla de color:** Sin colores vivos. La paleta es monocromática. El contenido (fotos, portadas) es la única fuente de color en la página.

### Typography
| Rol | Fuente | Tamaño | Peso |
|-----|--------|--------|------|
| Títulos / Display | Space Mono | 32–64px | 700 |
| Subtítulos / Labels | Space Mono | 14–24px | 400 |
| Body / Long text | IBM Plex Sans | 16–18px | 400 |
| Metadata / Técnico | Space Mono | 12–13px | 400 |
| Código | Space Mono | 14px | 400 |

**Rationale:** Space Mono para headings da el stark académico de los álbumes de referencia. IBM Plex Sans para body evita fatiga ocular en textos largos (bio, blog).

### Spacing & Layout
- Base: `4px` — escala: 4, 8, 12, 16, 24, 32, 48, 64, 96, 128
- Border radius: `0px` por defecto (sharp, sin redondeos) — excepción: `2px` en el progress bar del player
- Max content width: `1200px` (centrado con padding horizontal 24px mobile / 48px desktop)
- Breakpoints: `sm: 640`, `md: 768`, `lg: 1024`, `xl: 1280`

### Component Style
Sharp (sin border-radius). Sin sombras. Sin gradientes. Sin transiciones lentas. Las interacciones son rápidas (150ms ease-out). El hover es sutil: `bg-surface-hover`. El reproductor de audio tiene una visualización de barras CSS animadas (tipo espectro) que reacciona al estado play/pause.

## 8. Authentication & Authorization

El sitio es 100% público. No hay login para visitantes.

**Sanity Studio** tiene su propio sistema de auth (cuenta de Sanity/Google). Solo Sergio tiene acceso al Studio. La URL del Studio es `https://studio.sergioluque.com` (o `sergioluque.sanity.studio` como fallback).

**Webhook de revalidación:**
- Sanity → webhook POST a `/api/revalidate?secret=REVALIDATE_SECRET`
- El endpoint llama `revalidatePath('/', 'layout')` para regenerar todas las páginas
- El secret se guarda en `SANITY_WEBHOOK_SECRET` en Vercel

## 9. Build Order

**IMPORTANTE:** Cada paso debe estar funcional antes de pasar al siguiente. Usar `/frontend-design` y `/shadcn-ui` en los pasos indicados.

---

**Paso 1: Scaffolding del proyecto**
```bash
pnpm create next-app@latest sergioluque --typescript --tailwind --app --src-dir --import-alias "@/*"
cd sergioluque
pnpm add -D prettier eslint-config-prettier
```
- Configurar TypeScript en modo strict (`"strict": true` en tsconfig.json)
- Configurar Tailwind v4 con las CSS variables del design system en `src/styles/globals.css`
- Instalar shadcn/ui: `pnpm dlx shadcn@latest init` — tema Dark, base color Neutral
- Añadir componentes shadcn: `button`, `card`, `badge`, `separator`, `input`, `textarea`, `label`
- Configurar las fuentes en `app/layout.tsx`: Space Mono + IBM Plex Sans vía `next/font/google`

**Paso 2: Setup de next-intl (i18n)**
```bash
pnpm add next-intl
```
- Crear `src/messages/es.json` y `src/messages/en.json` con todos los strings de UI
- Crear `middleware.ts` con locale detection (default: `es`, supported: `['es', 'en']`)
- Crear `src/lib/i18n/request.ts`
- Reestructurar `app/` para tener `app/[locale]/`
- Verificar que `/` redirige a `/es/` y `/en/` funciona

**Paso 3: Setup de Sanity Studio**
```bash
pnpm create sanity@latest -- --project-name "sergioluque" --dataset production --template clean
```
- Crear todos los schemas en `sanity/schemas/`: `obra.ts`, `proyecto.ts`, `post.ts`, `evento.ts`, `publicacion.ts`, `settings.ts`
- Configurar el singleton de `settings` (no se puede crear más de uno)
- Crear `src/lib/sanity/client.ts` con `createClient({ projectId, dataset, apiVersion, useCdn: true })`
- Crear `src/lib/sanity/queries.ts` con todas las GROQ queries
- Crear `src/lib/sanity/image.ts` con `urlForImage` helper
- Deploy del Studio: `pnpm sanity deploy` → `studio.sergioluque.com`
- Cargar contenido de prueba: 3 obras, 1 proyecto, 1 post, settings con bio

**Paso 4: Cloudflare R2 para audio**
- Crear bucket `sergioluque-media` en Cloudflare R2
- Configurar como público con custom domain `media.sergioluque.com`
- Subir 2-3 archivos MP3 de prueba
- Actualizar las obras en Sanity con las URLs de audio: `https://media.sergioluque.com/audio/obra-slug.mp3`
- Documentar el proceso de subida en el README del proyecto

**Paso 5: Componentes base de layout**
Usar **`/frontend-design`** para este paso.
- `Header.tsx`: Logo (nombre "Sergio Luque" en Space Mono), navegación principal, `LanguageSwitcher`
- `Footer.tsx`: Links de navegación, email, año
- `LanguageSwitcher.tsx`: botón simple "ES / EN" que usa `usePathname` de next-intl para cambiar locale manteniendo la ruta actual
- Implementar el layout `app/[locale]/layout.tsx` que incluye Header y Footer

**Paso 6: AudioPlayer (componente crítico)**
Este es el componente más complejo del sitio. Construirlo aislado antes de usarlo en páginas.

Crear `src/components/audio/AudioPlayer.tsx` ("use client"):
- HTML5 `<audio>` element (oculto) con ref
- Estado: `isPlaying`, `currentTime`, `duration`, `isLoading`
- Controls: botón Play/Pause, barra de progreso clicable (seek), display de tiempo `MM:SS / MM:SS`
- `WaveformBars.tsx`: 30-40 barras CSS que animan con `animate-pulse` cuando `isPlaying === true` y se detienen en pause. Las alturas son estáticas (array hardcodeado de alturas aleatorias, generado una vez por obra).
- Diseño: fondo `--surface`, bordes `--border`, controles en `--text-primary`, barra de progreso en `--accent`
- Crear también `AudioPlayerMini.tsx`: versión compacta para los WorkCards en el catálogo

**Paso 7: Página de Obras (catálogo)**
Usar **`/shadcn-ui`** para los componentes de UI.
- `WorksPage`: Server Component que llama `getObras(locale)` de Sanity
- `WorkGrid`: grid de 2 columnas en desktop, 1 en mobile
- `WorkCard`: imagen (si existe), título, año, formación, duración + `AudioPlayerMini`
- Página individual `obras/[slug]`: `AudioPlayer` full width, metadata completa, descripción en Portable Text
- Implementar `PostBody.tsx` usando `@portabletext/react`

**Paso 8: Homepage**
- Hero: nombre completo + rol en Space Mono grande, minimal, ocupa el viewport
- Sección "Obras seleccionadas": 3-4 obras con `isFeatured: true` con AudioPlayerMini
- Sección "Próximos conciertos": los 3 eventos futuros más próximos
- Sin imágenes de fondo, sin gradientes — solo tipografía y espacio

**Paso 9: Bio + CV**
- `BioPage`: Portable Text renderizado con foto a la derecha en desktop
- Link de descarga del CV: `<a href={settings.cvPdfUrl} download>` — simplemente un link al PDF en R2
- Nada complejo aquí

**Paso 10: Proyectos**
- Grid de proyectos con imagen cover
- Página individual con galería de imágenes (Sanity Image CDN con srcset automático)

**Paso 11: Blog**
- Listado de posts con fecha, título, excerpt
- Página individual con Portable Text completo (usar el mismo `PostBody.tsx` creado en Paso 7)
- Paginación básica (o scroll infinito si se prefiere)

**Paso 12: Publicaciones académicas**
- Lista simple: título, journal, año, abstract corto, link/DOI
- No necesita página individual — todo visible en el listado

**Paso 13: Conciertos**
- Dos secciones: Próximos y Pasados (filtrado por fecha en GROQ)
- Cada evento: fecha, título, venue, ciudad, link externo

**Paso 14: Formulario de contacto**
- `ContactPage` con React Hook Form + Zod validation
- Server Action que llama Resend API
- Estados: idle → loading → success/error
- Campo oculto honeypot para spam básico

**Paso 15: SEO, sitemap y webhook**
- `metadata` dinámica en cada page (title, description, OG image)
- `app/[locale]/obras/[slug]/page.tsx`: `generateMetadata` con título de la obra
- `app/sitemap.ts`: genera sitemap con todas las URLs en ambos idiomas
- `app/[locale]/blog/feed.xml/route.ts`: RSS feed del blog
- Webhook de Sanity: `app/api/revalidate/route.ts` para ISR on-demand
- OG images: usar `next/og` (ImageResponse) para generar OG images dinámicas

**Paso 16: Deploy y configuración final**
```bash
vercel --prod
```
- Configurar variables de entorno en Vercel (ver Sección 10)
- Configurar DNS: `sergioluque.com` → Vercel, `studio.sergioluque.com` → Sanity, `media.sergioluque.com` → Cloudflare R2
- Activar webhook en Sanity Studio: Settings → API → Webhooks
- Lighthouse audit final: corregir cualquier issue de performance/a11y

## 12. Deployment Strategy

### Hosting
- **Frontend:** Vercel — deploy automático desde `main` branch. Preview deploys para PRs. ISR nativo.
- **Sanity Studio:** `pnpm sanity deploy` → se despliega en `sergioluque.sanity.studio` o custom domain `studio.sergioluque.com`
- **Audio/Media:** Cloudflare R2 con custom domain `media.sergioluque.com`

### CI/CD
- Push a `main` → Vercel despliega automáticamente
- Push a otras ramas → Preview deploy con URL única (útil para revisar cambios antes de publicar)
- Sanity Studio: deploy manual cuando se actualicen los schemas

### Domain & DNS
```
sergioluque.com          → A/CNAME a Vercel
```

### Revalidación de contenido (ISR on-demand)
Cuando Sergio publica o edita contenido en Sanity Studio, el webhook llama automáticamente a `/api/revalidate` que ejecuta `revalidatePath('/', 'layout')`, regenerando todas las páginas estáticas en Vercel. Esto significa que el sitio se actualiza en segundos sin rebuild.

## 16. Reglas No Negociables

1. **TypeScript strict** — no `any`, no `@ts-ignore` sin justificación
2. **Server Components por defecto** — "use client" solo en AudioPlayer, LanguageSwitcher, ContactForm
3. **CSS variables para colores** — nunca hex hardcodeados en componentes. Siempre usar las variables definidas en globals.css
4. **Audio player compatible con iOS Safari** — no autoplay. El audio solo empieza desde una interacción explícita del usuario. Testear en Safari antes de marcar como completo.
5. **GROQ queries centralizadas** — todas en `lib/sanity/queries.ts`. Las páginas no hacen fetch inline a Sanity.
6. **Sharp style** — border-radius 0px en toda la UI. Si tienes que agregar redondeo, justificarlo explícitamente.
7. **Sin colores vivos** — el diseño es monocromático. No agregar colores de marca, gradientes ni sombras decorativas.
8. **Bilingüe completo** — todo texto visible al usuario debe existir en `es.json` y `en.json` (strings UI) o en los campos localizados de Sanity (contenido). No dejar strings hardcodeados en los componentes.
